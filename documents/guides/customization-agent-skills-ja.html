1.Agent Skills とは

Agent Skills は、フォルダに格納された指示、スクリプト、およびリソースの集合です。GitHub Copilot が関連タスクを実行する際に自動的に読み込むことで、Copilot の能力を拡張できます。

Agent Skills はオープン標準で、VS Code の GitHub Copilot、GitHub Copilot CLI、GitHub Copilot コーディングエージェントなど、複数の AI エージェント間で活用できます。

1.1 Agent Skills の主な利点

  • Copilot を特化させる： 繰り返しのコンテキスト入力なしに、ドメイン固有のタスクに対応
  • 繰り返しを削減： 作成した Skill は、すべての会話で自動的に活用
  • 能力を組み合わせる： 複数の Skill を組み合わせることで、複雑なワークフローを構築
  • 効率的に読み込む： 関連するコンテンツだけが文脈に読み込まれるため、コンテキストウィンドウを効率的に利用

1.2 Agent Skills と カスタム指示の違い

項目 Agent Skills カスタム指示
目的 専門的な能力とワークフローを教える コーディング標準とガイドラインを定義
適用範囲 複数の AI ツール間で利用可能 VS Code と GitHub.com のみ
内容 指示、スクリプト、例、リソース 指示のみ
読み込み方式 タスク関連時にオンデマンド読み込み 常に適用（またはグロブパターン）
標準 オープン標準（agentskills.io） VS Code 固有

1.3 いつ Agent Skills を使うべきか

Agent Skills を用いる場面：

  • 複数の AI ツール間で再利用可能な能力を作成したい
  • スクリプト、例、またはその他のリソースを指示と一緒に含めたい
  • テスト実行、デバッグ、デプロイメント など、特殊なワークフローを定義したい
  • AI コミュニティと能力を共有したい

カスタム指示を用いる場面：

  • プロジェクト固有のコーディング標準を定義したい
  • 言語またはフレームワークの慣例を設定したい
  • コードレビューまたはコミットメッセージのガイドラインを指定したい
  • ファイルタイプに基づくルールを適用したい

2.Skill を作成する

Skill は、SKILL.md ファイルを含むディレクトリとして構成されます。VS Code は 2 つのタイプの Skill をサポートしています。

2.1 Skill の保存場所

Skill の種類 保存場所
プロジェクト Skill .github/skills/
.claude/skills/
.agents/skills/
個人 Skill ~/.copilot/skills/
~/.claude/skills/
~/.agents/skills/

カスタマイズ： chat.skillsLocations 設定を使用することで、プロジェクト Skill の追加の保存場所を構成できます。これは、異なるフォルダ構造で Skill を組織化する、または複数の Skill ディレクトリを持つ場合に有用です。

2.2 Skill を作成するための手順

  1. Chat ビューで「Configure Chat」（歯車アイコン）を選択して、Chat Customizations エディタを開きます。
  2. 「Skills」タブから、「New Skill (Workspace)」または「New Skill (User)」を選択します。Skill を保存する場所に応じて選択してください。
  3. Skill の保存場所を選択し、Skill の名前を入力します。
  4. YAML フロントマターを記入し、ファイルの本文に指示を追加して、SKILL.md ファイルを完成させます。
  5. （オプション）スクリプト、例、またはその他のリソースを Skill ディレクトリに追加します。

2.3 AI による Skill 生成

チャットで /create-skill を入力し、作成したい Skill の機能を説明することで、AI が Skill を生成できます。例えば：

  • 「統合テストを実行およびデバッグするための Skill」
  • 複雑な問題をデバッグした多段階プロセスから Skill を抽出する

Copilot は明確化に関する質問を行い、ディレクトリ構造、指示、およびフロントマターを備えた SKILL.md ファイルを生成します。

3.SKILL.md ファイルフォーマット

SKILL.md ファイルは、YAML フロントマターを備えたマークダウンファイルで、Skill のメタデータと動作を定義します。

3.1 ヘッダー（YAML フロントマター）

ヘッダーは --- で囲まれた YAML フロントマターであり、Skill のメタデータを定義します。ここで定義される情報が Copilot による Skill の検出と動作制御に直接影響します。

必須フィールド

フィールド 説明・ガイドライン 例
name
（必須）		 Skill の一意識別子。小文字のみ使用。スペースはハイフンに置き換え。最大 64 文字。

重要：親ディレクトリ名と完全に一致する必要があります。例えば、ディレクトリが skills/react-performance/ なら、name は react-performance である必要があります。一致していない場合、Skill は読み込まれません。

命名のコツ：動詞 + 対象（例：test-react-components, debug-api-calls, deploy-to-vercel） 		react-performance
test-integration
debug-api-calls
description
（必須）		 Skill の機能と用途についての説明。最大 1024 文字。

「何ができるのか」「どんな時に使うのか」を具体的に記述してください。Copilot はこの説明を読んで、ユーザーのリクエストに関連するかを判断し、自動的に読み込むかどうかを決定します。

効果的な説明の傍訳例：
❌ 「React コードを書く」
✅ 「React コンポーネントをテストするための Jest テストスイートを生成します。useContext、Props、State の各動作をテストします。/test-react-components [component file] で実行」 		「React のパフォーマンス最適化に関するベストプラクティスを適用します。不要な再レンダリングを防止し、メモ化戦略を提案します。」

オプショナルフィールド

フィールド 説明・デフォルト値 用途
argument-hint Skill をスラッシュコマンド（/my-skill）で呼び出す際、ユーザーに表示されるヒントテキスト。
形式：[引数1] [引数2] 		例：[test file] [coverage target]
ユーザーが / 入力時に表示され、何を指定すべきか分かりやすくなります。
user-invocable 此の Skill をスラッシュメニュー（/）に表示するか。デフォルト：true
false に設定すると、/ メニューから隠れます。ただ、Copilot は関連タスクで自動読み込みは可能です。 		背景知識 Skill（プロジェクト規約など）は false に設定し、自動読み込みのみにすることで、メニューをシンプルに保つ。
disable-model-invocation Copilot が関連性に基づいて自動読み込みするかを制御。デフォルト：false（自動読み込み有効）
true に設定すると、ユーザーが明示的にスラッシュコマンドで呼び出す場合のみ読み込まれます。 		特殊なコマンドのみ実行する Skill（例：/deploy-production）は true に設定し、誤った自動読み込みを防止。

3.2 本文（マークダウン形式の詳細指示）

YAML フロントマター（--- の後）の # My Skill 以降が本文です。ここに、Copilot が実行すべき具体的な指示を記述します。

本文に含めるべき要素（推奨順）：

  1. 大見出し（# タイトル）： Skill の名前を簡潔に記述
  2. 概要段落（1-2文）：「このスキルは何をするか」を１文で説明
  3. 使用場面（## When to Use）： どのような状況で使うべきかを列挙
  4. ステップバイステップ手順（## How to Use または ## Steps）： 実行すべき手順を番号付きリストで記述
  5. 具体例（## Examples）： 入力と出力のサンプルを示す
  6. ルール・ガイドライン（## Rules または ## Guidelines）： 従うべき基準や制約条件
  7. リソースへの参照（## Resources）： 含まれるスクリプトやテンプレートへのリンク

ファイル参照の方法：

Skill ディレクトリ内のスクリプトやリソースを参照する場合、相対パスを使用します。

[test template](./test-template.js) [deployment script](./scripts/deploy.sh) [configuration guide](./docs/config.md)

マークダウン形式のベストプラクティス：

  • 見出しレベルの一貫性： #（大）→ ##（中）→ ###（小）を正しく使用
  • コードブロック： ```javascript のように言語名を指定すると、シンタックスハイライトが有効になります。
  • リスト形式： 複数の選択肢や手順は箇条書き（- または番号 1.）で記述
  • 強調： 重要な部分は **太字** または _斜体_ で強調
  • 前後の文脈： 技術的な用語は最初に定義し、その後、既知として使用

3.3 完全な SKILL.md の実装例

例1：React テスティング Skill

--- name: react-component-testing description: Generate comprehensive Jest test suites for React components. Tests cover props, state, hooks (useState, useContext, useEffect), event handling, and edge cases. Use when you need to write robust tests for React components. argument-hint: [component file path] [test coverage %] user-invocable: true disable-model-invocation: false --- # React Component Testing Skill This skill generates production-ready Jest test suites for React components. ## When to Use - Writing unit tests for new React components - Adding test coverage to existing components - Testing complex component logic (hooks, state management) - Ensuring edge cases are covered ## How to Use 1. Specify the React component file to test 2. Define the desired test coverage percentage (e.g., 80%, 100%) 3. List any external dependencies or custom hooks used 4. The skill generates a complete test file with describe blocks and test cases ## Example Input ``` File: ./src/components/UserCard.jsx Coverage target: 85% Custom hooks: useUser, usePermissions ``` ## Example Output ```javascript import { render, screen, fireEvent } from '@testing-library/react'; import UserCard from './UserCard'; describe('UserCard Component', () => { test('renders user name correctly', () => { // Test implementation }); // More tests... }); ``` ## Guidelines - Always include tests for success and error paths - Use descriptive test names following "should..." pattern - Mock external API calls and dependencies - Include tests for user interactions (click, input, etc.) - Cover all conditional branches in the component ## Resources - [Jest Testing Framework](./docs/jest-guide.md) - [Testing Library Best Practices](./docs/testing-library.md) - [Example test template](./templates/test-template.js)

例2：バックグラウンド知識 Skill

--- name: react-project-coding-standards description: React プロジェクトの社内コーディング標準。命名規則、フォルダ構造、ベストプラクティスを定義します。すべての React コード生成時に自動的に適用されます。 user-invocable: false disable-model-invocation: false --- # React プロジェクト コーディング標準 このスキルは、すべての React コードが従うべき標準を定義します。 ## 命名規則 - **コンポーネント：** PascalCase（例：UserCard、LoginForm） - **カスタムフック：** camelCase で 'use' で始まる（例：useUser、useAuth） - **定数：** UPPER_SNAKE_CASE（例：API_URL、MAX_RETRIES） - **ファイル名：** エクスポート名と同じ ## フォルダ構成 ``` src/ ├── components/ # 再利用可能なReactコンポーネント ├── hooks/ # カスタムReactフック ├── utils/ # ユーティリティ関数 ├── types/ # TypeScript型定義 ├── services/ # API呼び出しとビジネスロジック └── constants/ # アプリケーション全体の定数 ``` ## React ベストプラクティス - **関数型コンポーネント使用：** クラス型ではなく、関数型コンポーネントと Hooks を使用 - **Props Drilling 回避：** 深いコンポーネント階層での状態共有には Context API を使用 - **単一責任の原則：** コンポーネントは 1 つの役割に絞る - **重い計算の最適化：** useMemo、useCallback でメモ化して不要な再計算を防ぐ - **キーの使用：** リスト要素にはインデックスではなく一意の ID を key として使用 - **エラーハンドリング：** Error Boundary でエラーキャッチし、フォールバック UI を提供

よくある間違いと対策：

  • ❌ ディレクトリ名と name が一致していない
    ✅ 解決：ディレクトリ名と SKILL.md の name を完全に一致させる
  • ❌ description が曖昧または短すぎる
    ✅ 解決：「何ができるか」「いつ使うか」を具体的に 1 段落で記述
  • ❌ 本文が指示ではなく、単なるドキュメント
    ✅ 解決：Copilot が実行すべきアクションを形式で記述（「...する」という命令形）
  • ❌ 大文字と小文字の混在
    ✅ 解決：YAML フロントマターのフィールド名はすべて小文字（user-invocable ✓、User-Invocable ✗）

4.Skill として活用する

Skill は、チャットの他のプロンプトファイルとともに、スラッシュコマンドとして使用できます。チャット入力フィールドで「/」を入力すると、利用可能な Skill とプロンプトのリストが表示されます。

4.1 スラッシュコマンドでの使用

Skill を呼び出すには、スラッシュコマンドを使用します。チャット入力フィールドで「/」を入力してください。

例えば：

  • /webapp-testing for the login page
  • /github-actions-debugging PR #42

4.2 Skill メニューの表示制御

user-invocable と disable-model-invocation を使用して、Skill へのアクセス方法を制御できます。

設定 / メニューに表示 自動読み込み 用途
デフォルト（未指定） はい はい 汎用 Skill
user-invocable: false いいえ はい バックグラウンド知識 Skill
disable-model-invocation: true はい いいえ オンデマンド Skill
両方設定 いいえ いいえ 無効 Skill

4.3 Copilot が Skill を活用する方法

Skill はコンテキストを効率的に保つため、段階的にコンテンツを読み込みます。以下に例を示します。

  1. 検出： Copilot は YAML フロントマターから Skill の名前と説明を読み込みます。「ログインページをテストしたい」と要求すると、Copilot はこれをwebapp-testing Skill にマッチングします。
  2. 指示読み込み： Copilot は SKILL.md の本文を文脈に読み込み、詳細なテスト手順とガイドラインにアクセスします。または、チャットで/webapp-testing を入力することで、この段階を直接トリガーできます。
  3. リソースアクセス： Copilot がテスト手順を実行する際、test-template.js やシナリオ例などの追加ファイルにアクセスします。ただし、参照されていないファイルは読み込まれません。

5.共有 Skill を活用する

他者が作成した Skill を活用して、Copilot の能力を拡張できます。github/awesome-copilot リポジトリには、Skill、カスタムエージェント、指示、およびプロンプトのコミュニティコレクションが含まれています。

5.1 共有 Skill のインストール手順

  1. リポジトリ内で利用可能な Skill を参照します。
  2. Skill ディレクトリを .github/skills/ フォルダにコピーします。
  3. SKILL.md ファイルを確認し、必要に応じてカスタマイズします。
  4. 必要に応じて、リソースを修正または追加します。

セキュリティに関する注意： 共有 Skill を使用する前に、必ず確認してください。これは要件とセキュリティ基準を満たしていることを確認するためです。

5.2 推奨される Skill リポジトリ

  • github/awesome-copilot： https://github.com/github/awesome-copilot

6.拡張機能から Skill を提供する

VS Code の拡張機能は、package.json の貢献ポイントを使用して、Skill を提供できます。

6.1 必要なフォルダ構造

extension-root/ └── skills/ └── my-skill/ # ディレクトリ名は SKILL.md の `name` フィールドと一致 └── SKILL.md # 必須

6.2 package.json での登録

拡張機能の package.json に、chatSkills コントリビューションポイントを追加します。path プロパティは対応する SKILL.md ファイルへのパスを指す必要があります。

{ "contributes": { "chatSkills": [ { "path": "./skills/my-skill/SKILL.md" } ] } }

重要： SKILL.md のフロントマターの name フィールドは、親ディレクトリ名と一致する必要があります。例えば、ディレクトリが skills/my-skill/ の場合、name 値は my-skill である必要があります。

A.YAML フロントマター詳細解説

YAML フロントマターは、マークダウンファイルの先頭に配置されるメタデータ定義ブロックです。ここに定義された情報は、プログラムやシステムが自動的に読み込み、ファイルの処理方法や表示方法を決定するために使用されます。

A.1 基本的な構成

YAML フロントマターは、常に次の形式で構成されます。

--- key1: value1 key2: value2 key3: value3 --- # ここからが本文 マークダウンコンテンツ...

各要素の説明：

  • 開始区切り： ---（3つのハイフン）— フロントマターの開始を示す
  • キー-バリューペア： key: value — メタデータの定義。コロン後に必ず空白が必要
  • 終了区切り： ---（3つのハイフン）— フロントマターの終了を示す
  • 本文： その後のマークダウンコンテンツ

A.2 YAML の記法ルール

1. キーと値の区切り

✅ 正しい形式 name: my-skill description: This is a description ❌ 間違った形式 name:my-skill # コロン後に空白がない name :my-skill # コロン前に空白がある name = my-skill # 等号記号を使用（YAML ではない）

2. 文字列値の引用符

✅ すべて有効 description: Simple description description: "Quoted description" description: 'Single quoted' ⚠️ 特殊文字を含む場合は引用符必須 description: "This has a: colon in it" description: "This has # a hash character"

3. ブール値（真偽値）

✅ すべて有効（true として認識） user-invocable: true user-invocable: True user-invocable: yes ✅ すべて有効（false として認識） user-invocable: false user-invocable: False user-invocable: no ❌ 引用符がないと文字列として認識される可能性 user-invocable: "true" # 文字列になる可能性がある

4. ネストされたリスト（配列）

✅ ハイフン形式 tags: - react - testing - automation ✅ ブラケット形式 tags: [react, testing, automation]

5. ネストされたオブジェクト

author: name: John Doe email: john@example.com organization: Company Name

A.3 Agent Skills での YAML フロントマター

必須フィールド：

フィールド 型 説明 例
name 文字列 Skill の一意識別子。小文字 + ハイフン。ディレクトリ名と完全一致。64文字以内。 react-component-testing
description 文字列 Skill の機能説明。「何ができるか」「いつ使うか」を明確に。1024文字以内。 "Jest テストスイートを生成します"

オプショナルフィールド：

フィールド 型 デフォルト 説明
argument-hint 文字列 （なし） スラッシュコマンド実行時のヒント表示。例：[file path] [options]
user-invocable ブール true / メニューに表示するか
disable-model-invocation ブール false Copilot の自動読み込みを無効化するか

実践例：

A.4 よくあるエラーと対処法

エラー 1: コロンの直後に空白がない

❌ エラー name:my-skill ✅ 正しい name: my-skill

エラー 2: 大文字と小文字の混在

❌ エラー Name: my-skill Description: Description here User-Invocable: true ✅ 正しい name: my-skill description: Description here user-invocable: true

エラー 3: ディレクトリ名と name が一致していない

❌ エラー（読み込まれない） ディレクトリ: skills/react-testing/ name: my-skill # ディレクトリ名と異なる ✅ 正しい ディレクトリ: skills/react-testing/ name: react-testing # ディレクトリ名と同じ

エラー 4: タブとスペースの混在

❌ エラー（YAML はスペースのみ） name: my-skill →description: Description # タブ文字を使用 ✅ 正しい（常にスペース 2 個） name: my-skill description: Nested description # スペース 2 個

エラー 5: フロントマターの区切りが不完全

❌ エラー（開始区切りがない） name: my-skill description: Description --- ✅ 正しい（開始と終了の両方が必要） --- name: my-skill description: Description ---

  • Online YAML Parser： https://yaml-online-parser.appspot.com/
